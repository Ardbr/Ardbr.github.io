在上一章节中，我们又实现了单个的 Transformer 块，在这一章节中，我们将会组装实现上图最左边的整个 Model 结构。

Tokenizer

Tokenizer（分词器） 是所有现代自然语言处理（NLP）模型（如 ChatGPT、Llama、BERT 等）的第一道大门。

计算机根本看不懂“苹果”或“Apple”这些字，它只认识数字。Tokenizer 的作用就是把一段文字切分成一个个小块（Token），然后把这些小块转换成计算机能识别的数字 ID。

📒Tokenizer 是人类语言（文本）和机器语言（数字）之间的“翻译官”。

为什么需要 Tokenizer

想象你要教一个只懂数学的机器人学英语：你不能直接给它看句子 “I love AI”。你需要把句子拆开，变成列表：[“I”, “love”, “AI”]。你需要一本字典，规定：“I” = 1, “love” = 2, “AI” = 3。最后你输入给机器人的其实是：[1, 2, 3]。

这个过程就是 Tokenization（分词）。

工作流程

第一步：切分

将一段长文本切成小的单元。注意分词器并不一定是按英语单词为最小划分单元，此处为了简便起见做了这个简化。

输入：Learning AI is fun!

输出：["Learning", "AI", "is", "fun", "!"]

第二步：映射

拿着切分好的小块，去查一个巨大的词表（Vocabulary）。这个词表是模型训练前就定好的，里面的每一行都有一个对应的编号。

  • "Learning" \(\rightarrow\) ID 4521
  • "AI" \(\rightarrow\) ID 123
  • "is" \(\rightarrow\) ID 11
  • "fun" \(\rightarrow\) ID 988
  • "!" \(\rightarrow\) ID 0

第三步：输出

将 ID 序列喂给神经网络模型。（也就是本次代码的输入就是 vocab_ids，形状为 [batch_size, seq_len]）

最终输入给模型的：[4521, 123, 11, 988, 0]

分词的三种流派

按单词分（Word-based）

英语单词变化太多了（run, running, runs, ran…），导致词表会变得超级大（几百万个）。如果遇到没见过的词（OOV），模型就傻了，通常会标记为 <UNK>（Unknown）。

按字符分（Character-based）

优点是词表很小（26 个字母+符号），不会遇到不认识的词。但是缺点是序列太长了，一句话可能变成几百个 ID，模型处理起来非常慢，而且单个字母没有语义（“a” 是什么意思？很难说）。

按子词分（Subword-based）—— 目前的主流（BPE）

这是现在 GPT、Llama 等模型使用的方法。它是前两者的折中方案。

原则：常见的词保持完整，罕见的词拆开。

  • 常见的词 "apple" \(\rightarrow\) ["apple"] (1 个 Token)
  • 罕见的词 "unfriendly" \(\rightarrow\) ["un", "friend", "ly"] (3 个 Token)

优点：词表大小适中（通常 3 万 ~ 10 万左右），既能保持语义，又能处理生僻词（拼凑出来），效率还高。

Model 代码实现

首先，我们会经过一个 Input Embedding 层，用来将 Tokenizer 分出的 ID 给升维嵌入成 hidden_dim 维度的向量。这里我们使用的是 nn.Embedding() 函数。

然后，我们经过 num_hidden_layers 数量的 Transformer Layer，由于我们实现好了 Block 块，这里直接调用 nn.ModuleList()，然后 for 循环放入 num_hidden_layers 个 Block 块即可。

之后经过 RMSNorm 进行一下归一化就可以了。

⚠️注意为了与 MiniMind 原项目尽可能保持一致，我们在这一章节中没有实现架构图中的 Linear 和 Softmax 这两层。这两层的作用是重新把 hidden_dim 维的向量映射成 vocab_size 个单词，并且使用 Softmax 给进一步转化成词库中每一个词的概率，从而决定我们应该选取哪一个词作为输出的下一个词。

第一点是 kv_cache 的最大长度 max_seq_length 是模型能够处理的“上下文窗口“的总长度，如果不考虑外推，这个长度限制了“输入的 Prompt“和“模型生成的回复“的总和。公式表达如下： \[ \text{self.freqs_cos 的长度} \ge \text{Prompt长度} + \text{生成的 Token 长度} \]

第一步：协同任务分解与分配

协同任务通过 \(\text{SMT}\) 算法分配给个体机器人（与之前论文相同）。

第二步：个体构建并预处理自动机

每个机器人整合协作任务，在本地利用拼接了协同任务的 \(\text{LTL}\) 表达式构造个体的自动机。之后每个机器人在自动机上利用 \(\text{BFS}\) 算法计算每个节点到接受节点的最近距离，作为后续每次选取节点的指标。

第三步：选择下一个节点转移

从自动机的候选节点（到接收节点的距离一样近）中综合考虑路径估计和协同等待任务，选取一个好的节点的转移条件（也就是需要完成什么任务）作为目标任务执行。 启发式算法衡量好节点的标准： \[ cost(q) = \underbrace{|SP_{p_i, p_\pi}|}_{\text{最短路径}} \underbrace{- \mathbb{I}[\pi \in \mathcal{S}_\text{colla}] \cdot r_\text{colla}}_{协同任务且正被等待}, q \in \mathcal{Q}_\text{cand} \] 也就是综合考虑当前到任务区域的距离（优先完成距离最近的任务），以及如果当前有协同任务正有机器人在等待了，这个任务也会提高优先级。

第四步：导航

现在我们有了当前机器人的位置，以及目标任务的位置，就可以使用未知环境下的点对点（Point-to-Point Navigation）算法搜索路径去执行，这里使用的是 \(\text{A*}\) 算法

第五步：更新地图，处理潜在任务

机器人可以通过雷达实时更新地图信息。并且可能在途中发现新的任务，新的任务会上传至中央服务器，中央服务器处理后下发给对应的机器人。

  • 如果是个体任务，直接下发给对应机器人，机器人拼接新的 \(\text{LTL}\) 表达式，构造新的自动机，恢复当前执行状态（记录之前执行的任务序列，也就是转移条件，在新的自动机上内部推演），继续执行。
  • 如果是协同任务，需要进行分解与分配，然后也下发给机器人群，每个机器人也按照上面的方法构造新的自动机，恢复状态，继续执行，但要通过强制制定顺序来避免发生死锁问题。
